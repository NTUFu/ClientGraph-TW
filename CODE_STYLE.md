## Code Style Guide

本檔定義「台灣上市櫃公司公開資料主管關係圖網站」的程式風格與實作準則。目的不是追求形式一致而已，而是降低資料處理、IndexedDB 交易、搜尋行為與 UI 實作的理解成本，避免多人協作時出現邏輯漂移。

**1. 通用原則**
1. 優先可讀性，其次才是技巧性寫法。
2. 先保證資料正確性，再做效能優化。
3. 單一函式只做一件主要事情。
4. 禁止為了簡短而犧牲語意。
5. 所有核心規則都要能被測試驗證。

**2. 命名規範**
1. 變數、函式使用 camelCase。
2. 元件檔名使用 PascalCase。
3. 常數使用 UPPER_SNAKE_CASE。
4. IndexedDB store 名稱使用小寫複數，例如 relationships、metadata。
5. 布林值命名使用 is、has、can 開頭，例如 isImporting、hasErrors。
6. 名稱必須帶語意，避免 a、b、tmp、data2 這類模糊命名。

**3. Vue 3 元件規範**
1. 一律使用 Composition API 與 script setup。
2. 每個元件先寫狀態，再寫計算值，再寫事件處理函式。
3. 避免在 template 內寫過長運算式，複雜邏輯應移到 script。
4. 與資料匯入、搜尋、IndexedDB 直接相關的邏輯應抽成可重用函式，不直接堆在元件內。
5. UI 文案集中管理，避免同一規則在多處以不同說法出現。

**4. JavaScript 實作規範**
1. 使用 const 優先，必要時才用 let。
2. 禁止使用 var。
3. 優先早返回，減少過深巢狀判斷。
4. 非同步一律使用 async、await，不混用 then 鏈式寫法。
5. 例外錯誤必須明確處理，不可靜默吞掉。
6. 需要保留原始資料時，不直接覆寫輸入物件。

**5. IndexedDB 規範**
1. 所有資料寫入都必須經過單一正規化流程。
2. 關鍵邏輯固定為 edgeKey upsert，不可使用 clear 重置資料。
3. transaction 作用範圍要小且清楚，避免在同一交易中混入無關邏輯。
4. 每次匯入都要輸出 inserted、updated、skipped、errors 統計。
5. metadata 更新必須與主資料寫入保持一致，不可只更新其中一方。
6. store 與 index 名稱固定後不得任意改名，若需更動必須同步設計 migration。

**6. 資料處理規範**
1. 原始公開資料欄位名稱盡量保留，避免過早重命名造成對照困難。
2. 派生欄位需明確標示用途，例如 normalizedSearch、sourceType、importedAt。
3. edgeKey 計算一律走共用函式，不可在不同模組各自拼接。
4. normalize 規則需固定：trim、全形空白轉半形、連續空白折疊。
5. 百分比與民國年月原始值預設保留字串格式，避免解析後失真。
6. 關係唯一鍵與圖形節點鍵必須分層：edgeKey 用於儲存層、personNodeKey 用於圖形層。
7. 圖形層預設採同名分離策略：personNodeKey = normalize(姓名) + | + normalize(公司代號)。
8. 若要改為跨公司同名合併，必須先提出決策紀錄與風險評估，再修改渲染規則。

**7. 搜尋與效能規範**
1. 搜尋優先使用 normalizedSearch 與 index，不直接對原始資料做重複拼接。
2. 先求結果正確，再做排序優化。
3. 大量資料處理要避免一次性 getAll 造成記憶體壓力。
4. 需要掃描資料時，優先使用 cursor。
5. 若效能優化改變行為，必須先定義驗證基準再調整。
6. 若使用 Cytoscape 或其他圖形引擎，重新渲染前必須清理舊實例與事件監聽。
7. 連續互動壓力測試需包含記憶體觀測，避免線性增長。

**8. 錯誤處理與觀測性**
1. 所有匯入錯誤都要能回報到檔名、資料列或欄位層級。
2. 使用者可理解的錯誤訊息與工程除錯訊息要分開。
3. 不可只回傳失敗，必須附帶原因。
4. 所有不可預期錯誤至少要有 console error 與 UI 提示。

**9. 註解與文件規範**
1. 註解只寫決策原因，不寫顯而易見的動作描述。
2. 複雜規則要寫在文件，不要只藏在程式碼。
3. 若邏輯對應開發計劃中的 Loop，註解或文件應能對回該 Loop。

**10. 測試規範**
1. edgeKey 計算、normalize、upsert、metadata 更新屬於必測核心。
2. 每次修正匯入邏輯都要回歸測試重複匯入與重疊覆蓋情境。
3. 搜尋功能至少要測姓名、公司代號、職稱三種查詢方式。
4. 修 bug 時優先補對應測試，避免同類問題再次發生。

**11. Git 與變更控制規範**
1. 一次提交只解一類問題，避免混入無關重構。
2. 修改資料模型、匯入規則、索引設計時，必須同步更新 DEVELOPMENT_PLAN.md 或相關文件。
3. 若變更影響 edgeKey 或 metadata 結構，需先記錄決策理由再實作。

**12. 本專案特別禁止事項**
1. 禁止在匯入流程中使用 store.clear()。
2. 禁止在多處實作不同版本的 edgeKey 規則。
3. 禁止將來源市場判定硬編碼在多個元件中。
4. 禁止未驗證就引入第三方 IndexedDB 封裝套件。

**13. 優先遵循順序**
1. 資料正確性。
2. 規格一致性。
3. 可維護性。
4. 使用者可理解性。
5. 效能優化。
