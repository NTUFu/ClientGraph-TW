## 主管關係圖網站開發計劃（Loop Engineering）

本計劃改採 Loop Engineering。核心不是先把所有功能一次做完，而是以短迭代回圈逐步降低不確定性。每一圈都包含四件事：定義假設、交付最小可驗證成果、執行量化驗證、依結果決定保留、修正或擴大。全案仍維持純前端、手動匯入、IndexedDB 增量合併、永不 clear 的前提。

**Loop Engineering 原則**
1. 每個 Loop 都要有明確假設。
2. 每個 Loop 只解一個主要風險，不同風險不要混在同一圈。
3. 每個 Loop 都有最小交付物，而不是半成品概念稿。
4. 每個 Loop 結束都要有可否決的驗證條件。
5. 驗證通過才進入下一圈；未通過就留在本圈修正。

**北極星目標**
1. 建立一個可離線、可多次匯入、可跨來源累積的主管關係資料底座。
2. 任何新檔匯入都只做 upsert，不可清空舊資料。
3. 使用者可在本機快速搜尋董事、法人代表人、公司與職稱關係。

**成功指標（KPI）**
1. 匯入 10,000 筆資料時間小於 3 秒。
2. 重複匯入相同檔案時 inserted = 0，updated 符合差異。
3. 50,000 筆資料規模下，關鍵字搜尋首屏回應小於 300ms。
4. 匯入錯誤可被定位到檔案、列資料、錯誤原因。
5. 重新整理頁面後資料仍完整存在 IndexedDB。

**固定範圍與邊界**
1. In Scope:
2. Vue 3 前端網站。
3. 原生 IndexedDB。
4. CSV/JSON 手動匯入。
5. edgeKey 增量合併。
6. 搜尋、篩選、匯入摘要。
7. Out of Scope:
8. 後端 API。
9. 自動抓取 data.gov.tw。
10. 關係圖進階視覺化引擎。
11. 雲端同步。

**正式資料來源**
1. 上市：https://data.gov.tw/dataset/22811
2. 上櫃：https://data.gov.tw/dataset/22812
3. 使用方式：由使用者先下載官方檔案，再於網站中手動匯入。

**核心規格不可變條件**
1. edgeKey = normalize(姓名) + | + normalize(公司代號) + | + normalize(職稱)
2. 匯入採 store.put() upsert。
3. 不可執行 store.clear()。
4. 同 edgeKey 新資料覆蓋舊資料；不同 edgeKey 與既有資料並存。
5. metadata 必須累積 latestDataMonth、sourceTypes、sourceSummary、importCount。

**Loop 0: 問題定義與資料契約凍結**
1. 假設：官方上市與上櫃資料可被統一為同一匯入資料契約。
2. 主要風險：欄位定義、型別、來源標記規則不清，導致後續反覆返工。
3. 最小交付：
4. 資料欄位字典。
5. edgeKey 規格文件。
6. 來源辨識規則（UI 選擇 > 檔名推斷 > unknown）。
7. metadata 欄位定義。
8. 驗證：
9. 能用上市、上櫃樣本各自映射為同一資料模型。
10. 能明確回答每個欄位保存原值或派生值的方式。
11. 退出條件：FRD 與資料字典簽核。

**Loop 1: 匯入管線最小可用版**
1. 假設：在不依賴第三方封裝的情況下，可用原生 FileReader + parser 穩定讀入 CSV/JSON。
2. 主要風險：檔案格式解析失敗或錯誤不可觀測。
3. 最小交付：
4. 檔案選取 UI。
5. JSON parser。
6. CSV parser。
7. 匯入前格式驗證與錯誤摘要。
8. 驗證：
9. 成功匯入官方下載檔。
10. 錯誤檔案可回報明確原因。
11. 退出條件：兩種格式都能完成 parse 並輸出標準 record 陣列。

**Loop 2: IndexedDB Schema 與增量合併核心**
1. 假設：以 relationships、metadata 兩個 store 即可支撐 MVP。
2. 主要風險：合併策略錯誤，造成資料被覆蓋遺失或重複堆積。
3. 最小交付：
4. relationships store，keyPath = edgeKey。
5. metadata store。
6. upsert 交易流程。
7. inserted、updated、skipped、errors 統計。
8. 驗證：
9. 第一次匯入 inserted > 0。
10. 第二次匯入重疊資料時 updated > 0。
11. 不重疊資料仍保留。
12. 任一流程都未發生 clear。
13. 退出條件：通過重複匯入、重疊覆蓋、重新整理持久化三個測試。

**Loop 3: 搜尋與索引效率回圈**
1. 假設：匯入時預先產生 normalizedSearch，即可讓 Cursor 搜尋在資料量提升時仍可接受。
2. 主要風險：搜尋效能差，或查詢邏輯與業務使用方式不匹配。
3. 最小交付：
4. normalizedSearch 欄位。
5. companyCode、name、title、dataYearMonth、sourceType、normalizedSearch index。
6. 關鍵字搜尋與市場、月別篩選交集。
7. 驗證：
8. 50,000 筆規模下首屏搜尋小於 300ms。
9. 姓名片段、公司代號、職稱三種查詢都可命中。
10. 退出條件：效能達標且搜尋結果排序可被業務接受。

**Loop 4: 使用者可理解性與操作安全回圈**
1. 假設：只要把來源、覆蓋規則、匯入結果說清楚，就能降低誤匯入與誤解風險。
2. 主要風險：使用者不知道自己匯入了什麼、覆蓋了什麼、資料是否混合。
3. 最小交付：
4. 匯入前來源選擇器。
5. 匯入後摘要面板（inserted、updated、skipped、errors）。
6. metadata 展示（單一來源、混合來源、最新資料年月）。
7. 匯入規則說明文案。
8. 驗證：
9. 測試使用者能正確說出本次匯入來源、市場、結果與覆蓋規則。
10. 退出條件：UAT 測試者無重大誤解，且錯誤率下降。

**Loop 5: 品質、相容性與效能硬化回圈**
1. 假設：在 MVP 成形後集中補強穩定性，比一開始全面優化更有效率。
2. 主要風險：大檔卡頓、Safari/Chrome 差異、錯誤追查困難。
3. 最小交付：
4. 分批匯入策略。
5. 交易失敗錯誤處理。
6. 跨瀏覽器驗證清單。
7. 匯入性能基準報告。
8. 驗證：
9. 10,000 筆匯入小於 3 秒。
10. Safari、Chrome、Edge 基本流程可用。
11. 退出條件：性能與相容性達標，無阻擋上線缺陷。

**Loop 6: UAT 與上線回圈**
1. 假設：只要前五圈輸出都可量化驗證，UAT 將聚焦在業務情境而非技術猜測。
2. 主要風險：最後驗收時才暴露業務理解差異。
3. 最小交付：
4. UAT 測試包（上市、上櫃、重疊樣本）。
5. 驗收腳本。
6. 問題分類與修正優先級。
7. 上線手冊。
8. 驗證：
9. 必測情境全部通過。
10. Critical、Major 缺陷為 0。
11. 退出條件：PM、SA、UAT 共同簽核。

**每個 Loop 的固定輸入輸出格式**
1. 輸入：本圈假設、要驗證的風險、測試資料、成功門檻。
2. 輸出：可執行成果、量化結果、發現問題、是否進入下一圈。
3. 決策：Keep、Revise、Expand。

**節奏與治理**
1. 每個 Loop 建議 3 到 5 個工作天。
2. 每圈結束開一次 Loop Review。
3. Review 只回答四件事：
4. 這圈假設是否成立。
5. 指標是否達標。
6. 是否有新的高風險未知項。
7. 下一圈是擴大還是修正。

**角色分工（Loop 版）**
1. PM：定義本圈商業目標、決定是否進下一圈。
2. SA：定義本圈假設、規格、驗證條件。
3. FE：實作本圈最小成果與量測。
4. QA：設計本圈測試案例與缺陷回饋。

**驗收與否決條件**
1. 任一 Loop 若無法量化驗證，視為未完成。
2. 任一 Loop 若需要靠人工口頭解釋才成立，視為規格不足。
3. 若發現 clear、重複資料暴增、來源不明、搜尋失效，立即退回前一圈修正。

**風險清單（Loop 導向）**
1. 欄位格式漂移：以容錯 parser 與 raw 保留緩解。
2. 大檔效能差：以分批 transaction 與進度回饋緩解。
3. 搜尋不準：以 normalizedSearch 與權重排序修正。
4. 使用者誤選來源：以 UI 明示與覆寫確認緩解。
5. 業務誤解覆蓋規則：以匯入摘要與 metadata 可視化緩解。

**文件交付物**
1. FRD。
2. SAD。
3. 資料字典。
4. Code Style Guide。
5. EditorConfig。
6. ESLint 設定。
7. Prettier 設定。
8. Loop Review 紀錄。
9. UAT 腳本與結果。
10. 上線手冊。

**下一步**
1. 先執行 Loop 0，凍結資料契約與 edgeKey 規格。
2. Loop 0 通過後才進 Loop 1，開始做匯入管線最小可用版。
3. 後續所有計畫異動都沿用本檔更新。
